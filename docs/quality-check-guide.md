# 出力品質チェックガイド

分析結果・レポート・PPTX・競合比較が実務で使えるレベルになっているかを確認するためのチェックリストです。
検証後に問題が見つかった場合は `docs/prompt-improvement-log.md` に記録してください。

---

## 1. 単体分析の品質

### 結論サマリー
- [ ] 抽象的な表現（「総じて好評」「一定の課題がある」）で終わっていないか
- [ ] 「この商品の強みは〇〇であり、課題は〇〇、次に取るべき打ち手は〇〇」という構造になっているか
- [ ] 業界名・商品特性を踏まえた具体的な言及があるか
- [ ] 200〜300文字程度で要点が伝わるか（長すぎ・短すぎに注意）

### 評価ポイント（rating_points）
- [ ] 「良い」「満足」など表層的なラベルではなく、何が良いのかが分かるか（例：「コスパの高さ」「翌日配送の速さ」）
- [ ] レビュー本文の実際の表現に根ざしているか（AI の創作になっていないか）
- [ ] 件数が多い順に並んでいるか
- [ ] クライアントが「確かにこれが評価されているんだ」と納得できるか

### 不満点（complaints）
- [ ] 「使いにくい」など曖昧なラベルではなく、具体的な不満内容になっているか（例：「説明書が分かりにくい」「サイズ展開が少ない」）
- [ ] FAQ・LP 改善・パッケージ見直しなど実際の改善行動に転用できるか
- [ ] 件数の少ない不満がノイズとして上位に来ていないか（過少評価に注意）

### 購買理由（purchase_reasons）
- [ ] 表層理由（「価格が手頃」「レビューが多かった」）と深層心理（「子どもに安全なものを使わせたかった」「失敗したくなかった」）が分かれているか
- [ ] クライアントが「この気持ちに訴求すれば刺さる」と感じられるか
- [ ] 出現件数が実態に近い数値になっているか（全件中の割合として違和感がないか）

### 訴求ワード（appeal_words）
- [ ] 広告コピーや LP のキャッチコピーにそのまま転用できる言葉か
- [ ] スコアの高いワードが本当に「強い言葉」になっているか（汎用的すぎるワードがトップに来ていないか）
- [ ] 業界固有の表現が含まれているか（コスメなら「うるおい」「毛穴」、教育なら「継続できる」「自信がついた」など）

### マーケティングインサイト（marketing_insights）
- [ ] priority: high のインサイトが本当に「重要度が高い」内容になっているか
- [ ] suggested_action が具体的な施策レベルになっているか（「LP を改善する」より「LP のファーストビューに〇〇の訴求を入れる」）
- [ ] high が多すぎて埋もれていないか（3〜5件程度が適切）

### 次アクション（actionItems）
- [ ] 「検討する」「改善する」といった動詞止めになっていないか
- [ ] 誰が・何を・どこで実行するか分かるか
- [ ] 分析期間・商品特性を踏まえたアクションになっているか

## 2. レポートページの品質

- [ ] タブ切り替えがスムーズに動作するか
- [ ] 各セクションのデータが欠損なく表示されているか（空欄・undefined などがないか）
- [ ] グラフ・バーの数値がデータと一致しているか
- [ ] 業界ベンチマークが下部に表示され、インサイトとの乖離がないか
- [ ] フィードバックフォームが表示され、保存できるか
- [ ] 「レポートを見る」→「1枚サマリー」→「PPTX出力」の導線が自然か
- [ ] 印刷プレビューで崩れないか（Chrome の印刷プレビューで確認）

---

## 3. 1枚サマリーの品質

- [ ] A4 縦1ページに収まっているか（スクロールが発生していないか）
- [ ] 「結論サマリー」「勝ち筋」「課題」「訴求ワード」「LP/広告メッセージ」「次アクション」の6ブロックが揃っているか
- [ ] 業界比較メモが表示されているか（unique / common の分類が正しいか）
- [ ] 文字が小さすぎず、老眼の方にも読めるフォントサイズか
- [ ] 印刷時に「no-print」要素が消えているか（戻るリンク・フィードバックフォームなど）
- [ ] クライアントに「このままミーティングで使えるか」と問われたときに Yes と言えるか

---

## 4. PPTX 出力の品質

### レイアウト
- [ ] スライド全体でフォントが統一されているか
- [ ] 日本語テキストの折り返しが崩れていないか（長い文が枠からはみ出していないか）
- [ ] 余白が十分あり、詰め込みすぎになっていないか
- [ ] カラーテーマが一貫しているか

### 内容
- [ ] 1スライド目（表紙）にプロジェクト名・業界・レビュー件数が入っているか
- [ ] 結論スライドだけ見て「何が分かったか」が伝わるか
- [ ] 勝ち筋 → 課題 → 次アクションの流れが自然か
- [ ] 訴求ワードが単なる羅列ではなく、「使える言葉」として見えるか
- [ ] 最終スライドに免責・調査概要が含まれているか

### 実用性
- [ ] クライアント（非エンジニア）にそのまま共有できるか
- [ ] ファイル名が分かりやすいか（`export.pptx` ではなく `プロジェクト名_分析結果.pptx` など）
- [ ] PowerPoint / Keynote で開いたときに崩れていないか（Google スライドでも確認）

---

## 5. 競合比較の品質

### 全体構造
- [ ] 単なる各商品の並列表示になっていないか
- [ ] 「A と B を比べたときに、C が勝てる市場」という示唆があるか
- [ ] 市場全体に共通する不満・ニーズが抽出されているか

### 比較インサイト
- [ ] 勝てる訴求（competitor A にはない・弱い点で自社が勝っている要素）が1件以上出ているか
- [ ] 各商品の強み・弱みが箇条書きではなく、示唆として書かれているか
- [ ] 「次に取るべき打ち手」が比較から導出されているか（単品分析の次アクションと違いがあるか）

### 比較 PPTX
- [ ] 商品ごとのスライドと、まとめスライドが分かれているか
- [ ] 「共通不満」「勝ちパターン」「推奨アクション」が明示されているか

---

## 6. 業界ベンチマークの品質

### 分類精度
- [ ] `common`（業界共通）と判定されたものが本当によく見られる傾向か
- [ ] `unique`（差別化）と判定されたものが、そのプロジェクト固有の特徴になっているか
- [ ] `emerging`（台頭中）の件数が適切か（多すぎると信頼性を損なう）

### 表現の安全性
- [ ] データ量が少ない（aggregated_insights に数件しかない）段階で `common` と断定していないか
- [ ] クライアントに見せたときに「これは本当ですか？」と問われない表現になっているか
- [ ] 解釈テキスト（interpretation）が過剰な約束をしていないか

### 空データ時の挙動
- [ ] `aggregated_insights` が 0 件のとき、適切な空状態メッセージが表示されるか
- [ ] ベンチマークカードが壊れた状態で表示されていないか

---

## 7. CSV / Markdown 出力の品質

### CSV エクスポート
- [ ] 文字コードが UTF-8 BOM 付きで、Excel で文字化けしないか
- [ ] 列ヘッダーが日本語で分かりやすいか
- [ ] 空値が `null` ではなく空文字や `—` で出力されているか
- [ ] 行数がレビュー件数と一致しているか

### Markdown エクスポート
- [ ] 見出し・箇条書きの構造が読みやすいか
- [ ] Obsidian / Notion に貼り付けて崩れないか
- [ ] コードブロックや特殊文字がエスケープされているか

---

## 8. Admin ログ・集計インサイトの品質

### /admin/usage
- [ ] 主要イベント（analysis_started / analysis_completed / feedback_submitted）がログに残っているか
- [ ] token_used が記録されており、コスト試算に使えるか
- [ ] エラーケースのログが `error` または `metadata` に詳細が残っているか

### /admin/insights
- [ ] 分析を実行するたびに件数が増えているか
- [ ] `confidence_score` が低いもの（データ不足）が適切にフラグされているか
- [ ] 同じラベルが重複登録されていないか（upsert が正しく動いているか）

### /admin/feedback
- [ ] レポート・one-pager からのフィードバックが一覧に表示されているか
- [ ] 低評価（overall_score ≦ 2）がハイライトされているか
- [ ] 改善メモの内容がプロンプト改善の参考になるか
